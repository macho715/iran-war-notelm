## 텔레그램 보고서 생성 흐름

---

### 전체 파이프라인

```
외부 뉴스 수집 → 중복 제거 → NotebookLM 업로드 → Phase 2 AI 분석 → 보고서 생성 → 텔레그램 전송
```

---

### 1단계: 뉴스 수집 (스크래핑)


| 소스     | 모듈                         | 방식                                                          |
| ------ | -------------------------- | ----------------------------------------------------------- |
| UAE 언론 | `scrapers/uae_media.py`    | Playwright + httpx (Gulf News, Khaleej Times, The National) |
| SNS    | `scrapers/social_media.py` | Facebook, Instagram 공개 검색                                   |
| RSS    | `scrapers/rss_feed.py`     | feedparser (Gulf News, Al Bawaba, BBC, AP 등)                |


각 기사는 `{source, title, link}` 형태의 dict로 수집됩니다.

---

### 2단계: 중복 제거

- `_filter_new()`: MD5 해시로 `_seen_hashes`와 비교해 신규 기사만 남김  
- `app.py` 358행: `new_articles = _filter_new(all_articles)`

---

### 3단계: NotebookLM 업로드 (선택)

- `_upload_to_notebooklm()`: 신규 기사를 텍스트로 묶어 NotebookLM에 업로드  
- 반환: `{notebook_id, source_id, notebook_url}`

---

### 4단계: Phase 2 AI 분석

- `_analyze_phase2()` → `analyze_with_notebooklm_or_fallback()`
- **1차**: NotebookLM에 JSON 형식 위협 분석 쿼리 → 파싱
- **2차**: 실패 시 `fallback_analyze()` (키워드 기반 규칙)

**결과 예시 (`AnalysisResult`):**

```python
{
  "threat_level": "LOW|MEDIUM|HIGH|CRITICAL",
  "threat_score": 0~100,
  "sentiment": "긴급|일반|회복",
  "abu_dhabi_level": "LOW|MEDIUM|HIGH|CRITICAL",
  "dubai_level": "LOW|MEDIUM|HIGH|CRITICAL",
  "summary": "...",
  "recommended_action": "...",
  "key_points": ["..."],
  "analysis_source": "notebooklm|fallback"
}
```

---

### 5단계: 보고서 텍스트 생성

- `reporter.build_report_text(articles, analysis, notebook_url)` → `_build_report()`

**구성 순서:**

1. **헤더**: `🚨 이란 전쟁 UAE 상황 실시간 보고 (YYYY-MM-DD HH:MM)`
2. **Phase 2 AI 평가** (analysis 있을 때):
  - 전체 위협 레벨·점수
  - 감성 톤
  - Abu Dhabi / Dubai 레벨
  - 요약, 권고, 핵심 포인트(최대 3개)
3. **도시별 기사** (`_split_by_city()`):
  - 1순위(아부다비): `abu dhabi`, `abudhabi`, `zayed`, `아부다비` 포함 기사 최대 3건
  - 2순위(두바이): `dubai`, `dxb`, `두바이` 포함 기사 최대 3건
  - 전체 뉴스: 그 외 기사 최대 5건
4. **안전 메시지**: `recommended_action` 또는 기본 문구
5. **출처 링크**: 기사 링크 + NotebookLM URL

---

### 6단계: 텔레그램 전송

- `send_report_text(report)` → `_send_telegram(report)`
- `_split_telegram_chunks()`: 4096자 초과 시 청크 분할
- 각 청크: Markdown 전송 시도 → 실패 시 `parse_mode=None`으로 재전송

---

### 코드 흐름 요약

```
app.hourly_job()
  → scrape_uae_media() + scrape_social_media() + scrape_rss()
  → _filter_new(all_articles)
  → _upload_to_notebooklm(new_articles)
  → _analyze_phase2(new_articles, notebook_context)
  → build_report_text(new_articles, analysis, notebook_url)   # reporter._build_report()
  → send_report_text(report_text)   # _send_telegram() + _send_whatsapp()
```

---

### 3단계 "선택" 표현

- 문서: "NotebookLM 업로드 (선택)"

- 실제: 신규 기사가 있으면 항상 업로드 시도. 실패 시 `notebook_context=None`이고 Phase 2는 fallback 사용.

- 판정: "선택"보다는 "시도(실패 시 fallback)"가 더 정확함.

### 권장 수정안

1. **코드 흐름 요약**에 즉시 경보·저장 단계 추가:

   ```

   → should_send_immediate_alert() → send_telegram_alert()  (HIGH/CRITICAL 시)

   → build_report_text(...)

   → send_report_text(report_text)   # Telegram + WhatsApp

   → *persist*storage(...)   # A+B 저장

   ```

2. **6단계** 제목을 "텔레그램 전송" → "텔레그램·WhatsApp 전송"으로 변경.

3. **3단계**를 "NotebookLM 업로드 (실패 시 fallback)" 정도로 수정.

4. **5단계 Phase 2** 항목에 `분석 소스(analysis_source)` 추가.

- 즉시 경보, WhatsApp, `persist_run`, `analysis_source` 등이 문서에 빠져 있어, 위 항목을 반영하면 더 정확해짐.
